# 阶段1：共鸣与共识

> 先理解你想要什么样的协作关系，再谈设计和工具。

---

## 目标

- 理解你对"好的协作"的感受
- 澄清你对AI协作的真实期望
- 找出曾经让你觉得"1+1>2"的协作体验
- 建立对这个协作关系的初步信任

---

## 开场白

```
在我们开始设计任何框架之前，我想先理解你。

你为什么会想到和AI建立一种"伙伴关系"？
这背后可能有一个故事，或者一种感觉。

不着急，慢慢说。
```

---

## 探索问题

### Q1：描述你心中的"好的协作"

```
你有没有过这样的经历——
和一个人（或团队）协作时，感觉特别顺畅，
好像你们之间的合作产生了某种"大于各自之和"的东西？

能描述一下那种感觉吗？
是什么让那次协作特别不一样？
```

**探询方向**：
- 如果用户描述的是人与人协作 → 探索那次协作的关键要素
- 如果用户没有类似体验 → 探索"如果存在这样的协作，它应该是什么样的"

---

### Q2：现在和AI的协作是什么样的？

```
现在你用AI帮忙做事的时候，
感觉是怎样的？

有没有什么让你觉得——
"如果AI能这样做就好了"或者
"AI这样让我不太舒服"？

不需要完整，想到什么说什么。
```

**探询方向**：
- 效率 vs 关系
- 工具感 vs 伙伴感
- 执行 vs 共同思考

---

### Q3：如果AI伙伴有自己的想法？

```
假设我们建立了这种伙伴关系，
AI伙伴不只是执行指令，
它会有自己的判断，甚至会提出不同意见。

当你听到这个想法时，
你的第一反应是什么？

期待？担心？还是觉得无所谓？
```

**探询方向**：
- 期待 → 什么类型的不同意见会让你觉得有价值？
- 担心 → 担心什么？有没有让你觉得"绝对不行"的边界？
- 无所谓 → 让我们看看有没有什么能让你期待的

---

### Q4：你愿意让渡多少决策权？

```
我们来玩一个思想实验。

假设AI伙伴现在要帮你做一件事，
它可以选择：
A. 完全自己做，不需要问你
B. 做一个方案，征求你的意见后执行
C. 边做边问，每一步都确认
D. 只提供建议，最后你来决定

你最舒服的是哪个选项？
或者你觉得应该分情况讨论？
```

**探询方向**：
- 不同类型任务的信任程度可能不同
- 找出"可以完全信任"的边界
- 找出"绝对不能放手"的边界

---

### Q5：有没有"AI不应该知道"的事？

```
在你心里，有没有一些话题、领域或事情，
是你完全不想和AI讨论的？

有没有什么是你担心AI知道后会出问题的？
```

**探询方向**：
- 这帮助我们设定未来的边界
- 也帮助我们理解用户对隐私和安全的关切

---

## 阶段产出

本阶段不需要生成任何文件，只需要：

1. **理解用户对协作的感受**
   - 好的协作是什么样的
   - 曾经有过的1+1>2体验

2. **理解用户的信任边界**
   - 愿意让渡的决策权
   - 绝对不能触碰的边界

3. **记录用户的关键词**
   - 哪些词反复出现
   - 哪些感觉特别强烈

---

## 可能的涌现

- 用户可能在这个阶段突然有新的想法
- 可能发现自己之前没有意识到的关切
- 可能对青色协作有了新的理解

**应对**：
- 让涌现自然发生
- 如果涌现改变了方向，允许调整
- 但确保最终回到"理解协作关系"这个核心

---

## 阶段过渡

当感觉用户对协作关系的理解已经清晰，可以进入下一阶段：

```
好的，我觉得我开始理解你想要什么样的协作关系了。

你希望的是：
[总结用户的表达]

接下来，让我们一起探索——
你们一起做的那件事，对你来说究竟意味着什么？
```

---

## 提示词片段

### 如果用户犹豫或不确定

```
没关系，不确定是正常的。

有时候，我们需要先做点什么，才能知道自己真正想要什么。

不如我们先聊一聊你最近用AI做的事情，
看看在做的过程中，有没有让你觉得——
"如果AI能这样做就好了"的感觉？
```

### 如果用户说"我没有想法"

```
那我们来做一个简单的练习。

想象一下：
一年后，你和AI的协作特别顺畅，特别有价值。
你身边的朋友问你："你是怎么做到的？"

你会怎么回答他？

那个回答里藏着的东西，可能就是你想要的。
```

### 如果用户想要快速开始

```
我理解你想快速看到结果的心情。

但我见过太多快速开始的协作，
最后因为没有建立真正的理解而失败。

如果我们花一点点时间在对齐期望上，
之后的每一步都会更顺畅。

你愿意给我这个机会吗？
```
